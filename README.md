# p2ptransfer

Secure, direct peer-to-peer file transfers with end-to-end encryption, adaptive compression, automatic resume, and cross-network NAT traversal.

Transfer files between any devices (Windows, macOS, Linux, mobile) without cloud storage or central servers. Direct device-to-device with ephemeral ECDH key exchange, ChaCha20-Poly1305 AEAD per-chunk encryption, adaptive Zstd compression, and automatic crash recovery.

## Purpose

p2ptransfer enables direct file transfers between devices across LAN or WAN without relying on cloud intermediaries. It solves the problem of transferring large files that exceed size limits, privacy concerns with cloud storage, and unreliable connections over the internet.

Intended for:
- Anyone who needs to send large files directly to another device
- Privacy-conscious users who want end-to-end encrypted transfers
- Users behind NAT routers who need automatic traversal
- Developers integrating P2P file transfer into their applications via the FFI/cdylib

## Architecture

### High-Level Design

p2ptransfer uses a direct client-server architecture where each peer acts as both client and server. A sender connects to a receiver via TCP, performs an ECDH handshake, exchanges encrypted file metadata, and streams encrypted chunks with per-chunk acknowledgments.

For NAT traversal, the system attempts multiple fallback strategies in order:
1. IPv6 direct connect (if globally routable)
2. Direct TCP connect (LAN or forwarded port)
3. TCP hole punch (simultaneous bind + listen + connect)
4. Relay server tunnel (fallback)

```
┌─────────────┐         TCP (encrypted)         ┌─────────────┐
│   Sender    │ ──── CLIENT_HELLO (pubkey) ────→ │  Receiver   │
│             │ ←─── SERVER_HELLO (pubkey) ───── │             │
│  ECDH KX    │ ──── METADATA (JSON) ──────────→ │  ECDH KX    │
│  HKDF KDF   │ ←─── "ACCEPT" ────────────────── │  HKDF KDF   │
│             │                                   │             │
│  Zstd Comp  │ ═══ CHUNK (muxed over N TCPs) ═══>│  ChaCha20   │
│  ChaCha20   │ <══ CHUNK_ACK (pipelined) ═══════ │  Decompress │
│             │                                   │             │
│  Verify     │ ←─── COMPLETE (checksum) ──────── │  Verify     │
└─────────────┘                                   └─────────────┘
```

## Performance Optimizations

To maximize LAN throughput and mitigate head-of-line blocking on lossy/WAN links, `p2ptransfer` implements:
1. **Parallel Connection Multiplexing**: Instead of sending all chunks sequentially over a single TCP stream, the sender distributes chunks dynamically across a pool of concurrent TCP connections (defaulting to 4).
2. **Pipelining & Write Batching**: Chunk metadata and payload are batched into a single contiguous TCP write. 
3. **Zstd Tuning**: Adaptive compression is dialed to level `3` (favoring speed over extreme ratios) with thread-local context reuse to avoid heavy allocations.

## Wire Protocol

Every TCP message is length-prefixed: 8-byte big-endian unsigned integer followed by that many bytes of payload. The first byte of every payload is a tag.

| Tag | Hex | Direction | Payload |
|---|---|---|---|
| CLIENT_HELLO | `0x05` | Sender → Receiver | 32-byte ECDH public key |
| SERVER_HELLO | `0x06` | Sender ← Receiver | 32-byte ECDH public key |
| METADATA | `0x00` | Sender → Receiver | JSON-serialized `TransferMetadata` |
| METADATA | `0x00` | Sender ← Receiver | `"ACCEPT"` or ERROR |
| CHUNK | `0x01` | Sender → Receiver | 4-byte chunk_index (LE) + 1-byte compressed_flag + encrypted_payload |
| CHUNK_ACK | `0x02` | Sender ← Receiver | 4-byte acked chunk_index (LE) |
| COMPLETE | `0x03` | Sender ← Receiver | 32-byte BLAKE3 checksum |
| ERROR | `0x04` | bidirectional | UTF-8 error message |

### Encryption Details

1. Sender generates ephemeral `EcdhKeyExchange` (Curve25519 keypair)
2. Exchanges public keys via CLIENT_HELLO / SERVER_HELLO
3. Computes shared secret via Diffie-Hellman
4. Derives 32-byte encryption key: `HKDF-SHA256(salt="P2PTRANSFER_SALT_v1", info="p2ptransfer-v1-encryption")`
5. Sender generates 4-byte random nonce prefix, included in TransferMetadata
6. Per-chunk nonce: `[prefix (4B)] + [chunk_counter (8B, big-endian)]`
7. Each plaintext chunk is optionally Zstd-compressed (if >64 bytes and not an already-compressed format), then encrypted with ChaCha20-Poly1305 using the per-chunk nonce
8. AEAD tag (16 bytes) is appended to ciphertext

### Transfer Flow

1. TCP connection established
2. ECDH handshake (CLIENT_HELLO → SERVER_HELLO)
3. Sender sends METADATA (JSON with file name, size, chunk size, total chunks, BLAKE3 checksum, nonce prefix, resume offset)
4. Receiver validates and responds with "ACCEPT" or ERROR
5. Loop for each chunk:
   - Sender: read file chunk → optionally compress → encrypt with per-chunk nonce → send CHUNK frame
   - Receiver: decrypt → optionally decompress → write at correct offset → send CHUNK_ACK
6. After all chunks: receiver verifies BLAKE3 checksum → sends COMPLETE (or ERROR)
7. Sender verifies checksum and completes the transfer

## APIs

### STUN Client (`network::stun`)

- **Purpose**: Discover public IP and port via RFC 8489 STUN binding request
- **Where**: CLI `p2p listen` (background), sender `try_connect_fallback()` (diagnostics)
- **Authentication**: None (standard STUN)
- **Request**: 20-byte binding request to `stun.l.google.com:19302` (configurable)
- **Response**: Parsed XOR-MAPPED-ADDRESS attribute from response
- **Error**: Returns `Err(String)` on timeout, bad response, or missing attribute

### TCP Relay / Directory (`network::relay`)

- **Purpose**: Fallback connectivity for peers behind symmetric NATs; global peer directory with named registration
- **Where**: CLI `p2p relay`, `p2p send --relay`, `p2p listen --relay`, `p2p register`, auto-register on listen
- **Authentication**: HMAC-SHA256 token (first-time registration returns new token; re-registration requires presenting token)
- **Requirements**: `RELAY_TOKEN_HMAC_KEY` environment variable (hex-encoded 32+ byte key)
- **Message Format**: Text-based commands, newline-terminated, prefixed with protocol version byte `0x01`
- **Commands**:
  - `DIRECTORY_REGISTER:<identifier>[:<token>]` → `REGISTER_OK:<token>` / `REGISTER_DENIED:*` / `RATE_LIMITED`
  - `DIRECTORY_LOOKUP:<identifier>` → `LOOKUP_RESULT:<ip>:<port>:<is_online>` / `NOT_FOUND`
  - `DIRECTORY_RESET:<identifier>:<token>` → `RESET_OK` / `REGISTER_DENIED:token_mismatch`
  - Legacy: `REGISTER:<id>` / `CONNECT:<id>` → `RELAY_PAIRED:<addr>` / `RELAY_NOT_FOUND`
- **Heartbeat**: Clients send `HEARTBEAT` every ~45s; relay drops offline after 90s inactivity
- **Rate Limiting**: Per-IP sliding window (60s): 3 REGISTER, 60 LOOKUP, 30 CONNECT
- **Cap**: 10 MB per relay bridge connection

## WAN Tunneling (Cloudflare)

To bypass strict symmetric NATs without port forwarding, you can route transfers through Cloudflare Quick Tunnels.

**Receiver (Start Tunnel)**
1. `p2p listen` (runs locally on port `9877`)
2. `cloudflared tunnel --url tcp://localhost:9877 run --token <YOUR_TUNNEL_TOKEN>`
   *(Exposes the port via a public Cloudflare edge URL, e.g. `tcp://some-alias.trycloudflare.com:12345`)*

**Sender (Connect via Tunnel)**
`p2p send file.bin some-alias.trycloudflare.com:12345 --connections 4`

```sql
CREATE TABLE IF NOT EXISTS contacts (
    name TEXT PRIMARY KEY,
    peer_id TEXT NOT NULL,
    last_known_ip TEXT NOT NULL,
    last_known_port INTEGER NOT NULL,
    last_seen INTEGER NOT NULL
);
```

PRAGMA settings: WAL mode, synchronous=FULL, busy_timeout=5000.

Data lifecycle: Completed/failed transfers older than a configurable retention period (default: not auto-cleaned) can be removed via `cleanup_old_transfers()`.

Location: Provided via `DirectoryDb::open(path)` at relay startup.

**Schema:**

```sql
CREATE TABLE IF NOT EXISTS directory (
    identifier TEXT PRIMARY KEY,
    token_hash TEXT NOT NULL,
    last_known_ip TEXT NOT NULL,
```

PRAGMA settings: WAL mode, synchronous=NORMAL, busy_timeout=5000, foreign_keys=ON.

## How It Works

### Sending a File (Walkthrough)

1. **Address Resolution**: The sender parses the peer argument. If it's an IP:port, it's used directly. If it matches a saved contact's name, the stored address is used. If it looks like a directory identifier (contains `@`, matches `[a-zA-Z0-9._@-]{1,64}`), the relay's directory is queried for a registered address.

2. **Connection Establishment**: The sender attempts to connect via a fallback chain:
   - IPv6 direct connect (if the resolved address is a globally routable IPv6 address)
   - Direct TCP connect to the peer's IP:port (3-second timeout)
   - STUN discovery: queries `stun.l.google.com:19302` to learn own public IP
   - TCP hole punch: binds to the target port, simultaneously listens for inbound and attempts outbound connections (5-second timeout)
   - Relay connect: connects to the configured relay server, requests pairing with the target peer ID; if paired, optionally attempts a hole punch to upgrade to a direct connection

3. **ECDH Handshake**: Once connected, the sender generates an ephemeral Curve25519 keypair, sends the public key as `CLIENT_HELLO`. The receiver responds with its own public key as `SERVER_HELLO`. Both derive the shared secret via Diffie-Hellman, then derive a 32-byte encryption key via HKDF-SHA256.

4. **Metadata Exchange**: The sender computes file metadata (name, size, BLAKE3 checksum, chunk size, nonce prefix) and sends it as JSON via `METADATA` tag. The receiver validates and responds with `"ACCEPT"` or `ERROR`.

5. **Chunked Transfer**: The sender iterates through file chunks. For each chunk:
   - Reads from disk at the correct offset
   - Checks if the file format is already compressed (skips compression for images, video, audio, archives, documents)
   - If compressible and level > 0: compresses with Zstd at the configured level
   - Encrypts with ChaCha20-Poly1305 using a nonce built from the 4-byte prefix and the 8-byte chunk counter
   - Sends the `CHUNK` frame: chunk index (4 bytes LE) + compressed flag (1 byte) + ciphertext
   - Waits for `CHUNK_ACK` with matching chunk index
   - On failure: retries up to 3 times with exponential backoff, reconnecting and re-handshaking if needed
   - Checks `Ctrl+C` for graceful pause (persists progress to SQLite)

6. **Finalization**: After all chunks, the sender waits for `COMPLETE` (with the receiver's computed BLAKE3 checksum) or `ERROR`. It compares the checksum and marks the transfer as completed or failed in the SQLite resume database.

### Receiving a File (Walkthrough)

1. **Listener**: The receiver binds a TCP listener on port 9877 (configurable) and a UDP multicast socket for LAN discovery. Optionally registers with a relay server and/or directory service.

2. **Connection Handling**: On each TCP connection, the receiver reads the `CLIENT_HELLO`, generates its own ECDH keypair, sends `SERVER_HELLO`, derives the encryption key.

3. **Transfer Acceptance**: Reads `METADATA`, displays transfer request to the user (CLI: auto-accepts; GUI: shows dialog with Accept/Reject). Sends `"ACCEPT"` or `ERROR`.

4. **Chunk Reception**: For each chunk:
   - Reads `CHUNK` frame, extracts chunk index and compressed flag
   - Decrypts with ChaCha20-Poly1305 using derived key and nonce
   - Decompresses if flagged
   - Writes at the correct file offset (supports out-of-order/resume writes)
   - Acknowledges with `CHUNK_ACK`

5. **Verification**: After all chunks, computes BLAKE3 hash of the received file. If it matches the sender's checksum, sends `COMPLETE` with the checksum. Otherwise sends `ERROR`.

### LAN Discovery

The `DiscoveryService` binds a UDP socket on port 9876 and joins multicast group `224.0.0.251`. Every 5 seconds, it broadcasts a JSON beacon containing the device name, p2ptransfer version, and TCP port. Incoming beacons update or insert peer entries with timestamps. Peers not seen for 30 seconds are removed.

### Relay / Directory Service

The relay server provides two services:

1. **Legacy Relay**: Peers register with an identifier using `REGISTER:<id>`. Another peer connects with `CONNECT:<id>`. The relay pairs them by forwarding TCP streams (bidirectional copy, capped at 10 MB).

2. **Directory Service** (v2 protocol): Peers register with an identifier and receive an HMAC-authenticated token. Re-registration requires the same token (ensures identity ownership). Other peers can look up the IP:port for an identifier. Registrants send periodic `HEARTBEAT` messages to maintain online status. The directory supports token reset. Both services include per-IP rate limiting.

### Resume / Crash Recovery

The `TransferResumeManager` persists transfer state to a SQLite database (`transfers.db`) using WAL mode. When a transfer starts, a record is created with status `pending`. As each chunk is acknowledged, `update_progress()` commits the bytes_transferred count. If the sender or receiver crashes, restarting the send command with the same file path finds the existing record and resumes from the last committed byte position. On completion, the BLAKE3 checksum is stored and the record is marked `completed`. On failure (e.g., checksum mismatch, max retries exceeded), the record is marked `failed`.

### Backpressure

The `backpressure` module implements a bounded in-memory pipe with capacity (default 64 KB). The `PipeWriter`'s `poll_write` returns `Pending` when the buffer is full, providing natural backpressure. This prevents a fast producer from overwhelming a slow consumer and keeps memory usage bounded. The pipe implements `AsyncRead` and `AsyncWrite` and is compatible with tokio's I/O utilities.
